# WitWeb 项目评估与更新计划

## 1. 项目概览

WitWeb 是一个基于 Next.js App Router 的全栈单体应用，主体代码位于 `web/`。项目当前并不是单一博客站，而是一个组合型产品，包含以下核心业务：

- 内容社区：文章、分类、评论、点赞/点踩、收藏
- 用户关系：个人主页、关注/粉丝、好友页
- 社交互动：私信会话、消息通知
- 管理后台：用户管理、文章管理、分类管理、统计与审计
- Studio：视频生成、AI Agent 写作、选题雷达

当前技术栈以 TypeScript 为主，前端采用 Next.js + React，后端 API 直接使用 Next.js Route Handlers，数据库统一为 PostgreSQL。

## 2. 当前架构理解

### 2.1 目录分层

- `web/src/app`
  页面路由和 API 路由入口
- `web/src/features`
  业务域页面组件，按 blog、admin、messages、user 等拆分
- `web/src/components`
  通用组件和 Studio 组件
- `web/src/lib`
  核心能力层，包括认证、JWT、配置、安全、仓储、Agent、Radar
- `web/src/lib/repositories`
  数据访问层，封装 PostgreSQL 查询
- `web/migrations`
  SQL 迁移文件
- `docs/`
  项目文档

### 2.2 典型调用链

项目的典型请求流程如下：

1. 页面组件发起 `fetch("/api/*")`
2. API Route 进行鉴权、参数校验、错误包装
3. 业务逻辑调用 repository
4. repository 使用 PostgreSQL 执行查询
5. API 返回统一格式响应

这种结构比“页面直接查库”更清晰，但仍属于单体应用内的紧耦合分层，而不是独立服务架构。

### 2.3 数据模型特征

数据库采用单库多业务模式，一个 PostgreSQL 实例承载所有主要模块：

- 社区内容：`users`、`posts`、`comments`、`likes`、`favorites`
- 社交关系：`follows`、`conversations`、`private_messages`
- 运营能力：`categories`、`friend_links`、`site_visits`
- Studio：`video_tasks`、`video_results`、`characters`
- Agent：`agent_runs`、`agent_steps`、`agent_artifacts`
- Radar：热点源、提醒规则、通知日志等

该模式对中小型产品开发效率较高，但随着业务增长，库内耦合和表关系复杂度会继续增加。

## 3. 项目优点

### 3.1 业务完整度高

项目已经具备较完整的产品能力，不是 Demo，也不是单点功能页。社区、后台、消息、Studio 四条业务线都已具备独立价值。

### 3.2 代码分层基本清晰

虽然仍有改进空间，但整体上已经形成以下边界：

- 页面/UI 层
- API 层
- repository 数据访问层
- 基础设施层

这比典型的“所有逻辑散落在页面和接口文件”要更容易维护。

### 3.3 数据库已统一到 PostgreSQL

项目已经从旧的 SQLite 方案迁移到 PostgreSQL，并保留了迁移脚本和 SQL migration 机制。这是一个重要的稳定性提升点。

### 3.4 有基础安全意识

项目已具备以下基础能力：

- JWT 认证
- 中间件鉴权
- RBAC 角色判断
- 敏感配置加密存储
- 基础环境变量约束

这说明项目不是完全无约束的快速堆叠代码。

### 3.5 Studio 模块有产品差异化

视频生成、AI Agent、选题雷达三块能力，使项目具备明显区别于普通博客系统的扩展方向。

## 4. 项目缺点

### 4.1 测试覆盖面不足

当前已有 Vitest 基础测试，但覆盖范围偏窄，主要集中在少量 admin API、auth 和基础组件上。

主要问题：

- 缺少博客主流程测试
- 缺少私信系统测试
- 缺少 Studio 关键链路测试
- 缺少端到端测试
- 缺少数据库行为级测试

这会直接影响重构安全性和版本迭代速度。

### 4.2 数据访问层高度依赖手写 SQL

目前项目直接基于 `pg` 和手写 SQL 实现 repository。

优点：

- SQL 可控
- 性能透明
- 无 ORM 额外抽象成本

缺点：

- 查询维护成本较高
- 类型推导能力有限
- 字段变更时重构成本高
- 大型 repository 文件容易膨胀

项目并非没有迁移系统，但缺少 ORM 或类型化查询工具带来的 schema 联动能力。

### 4.3 状态管理与数据获取策略原始

当前前端主要依赖：

- React Context
- `localStorage`
- 组件内 `fetch + useEffect`
- 简单缓存工具

主要问题：

- 认证状态依赖浏览器存储，安全边界一般
- 服务端状态和客户端状态混用
- 请求缓存、失效、重试、去重不统一
- 页面组件中存在较多重复请求逻辑

### 4.4 错误处理在前端仍较分散

后端 API 已有统一错误包装雏形，但前端组件中的错误处理仍较分散：

- 多处重复 `try/catch`
- 提示文案不统一
- 缺少统一日志上报
- 缺少真正落地的监控链路

这会导致线上问题难以排查。

### 4.5 API 路由数量多且组织偏碎

当前 API 路由较多，按资源拆分本身并不是问题，但目前存在以下现象：

- 业务域下接口粒度过细
- 横切逻辑抽象不足
- 不利于后续统一文档和权限治理

短期问题不是“路由文件多”，而是“结构分散、重复模式较多”。

### 4.6 文档体系不完整

当前文档更偏部署和安全说明，缺少面向维护者的核心文档：

- 架构说明
- 模块边界说明
- API 文档
- 数据模型说明
- 组件使用说明
- Studio 外部依赖配置文档

这会显著增加新成员接手成本。

### 4.7 依赖治理机制不足

项目已有 lock 文件，但依赖治理仍不够完整：

- 存在范围版本依赖
- 缺少依赖升级策略
- 缺少 lockfile 一致性校验
- 缺少 CI 层面的安装一致性约束

这会导致环境漂移风险。

## 5. 优先级判断

当前阶段最关键的问题排序建议如下：

1. 测试覆盖不足
2. 认证与状态管理不稳
3. 错误处理和监控不统一
4. repository 与 API 结构逐渐膨胀
5. 文档不足
6. 数据层类型化能力不足
7. 依赖治理机制不完善

不建议当前直接把“引入 ORM”作为第一优先级。相比之下，测试、认证、错误处理和结构收敛更紧迫。

## 6. 更新目标

未来迭代的目标应分为四类：

- 提升稳定性
- 提升安全性
- 提升可维护性
- 为 Studio 和社区功能继续扩展留出结构空间

## 7. 分阶段更新计划

### 第一阶段：质量兜底

目标：先让核心链路可验证、可重构。

计划：

1. 为以下核心链路补单元测试和 API 测试
   - 登录注册
   - 文章列表与文章发布
   - 私信发送与会话读取
   - Agent run 创建与详情读取
2. 引入 E2E 测试框架，优先使用 Playwright
3. 建立最小回归集
   - 登录
   - 发布文章
   - 私信发送
   - 后台登录
4. 对关键 API 添加 request id、失败日志和可追踪错误信息

建议周期：1 到 2 周

### 第二阶段：认证与数据获取收敛

目标：减少前端状态混乱和登录态风险。

计划：

1. 逐步减少 `localStorage` 对认证态的依赖
2. 评估并引入统一服务端状态管理方案
   - React Query 或 SWR
3. 统一列表页、详情页、消息页的数据获取模式
4. 引入统一错误提示和页面级 Error Boundary
5. 为请求失败、鉴权失败、权限不足建立一致反馈

建议周期：1 到 2 周

### 第三阶段：后端结构整理

目标：降低单体内部的复杂度。

计划：

1. 拆分过长的 repository 文件
2. 按业务域梳理 `video`、`agent`、`radar` 的接口结构
3. 抽出共享 API 处理逻辑
   - 鉴权
   - 分页
   - 审计
   - 错误映射
4. 统一业务类型定义，减少页面层重复声明接口结构
5. 补充模块级文档

建议周期：2 到 4 周

### 第四阶段：数据层升级评估

目标：在不破坏现有能力的前提下提升类型化能力。

计划：

1. 评估是否引入类型化查询工具
   - 优先考虑 Kysely 或 Drizzle
2. 不建议全量一次性迁移
3. 先在新模块或低风险查询中试点
4. 若收益明确，再逐步迁移旧 repository

建议周期：2 到 6 周，可与第三阶段部分并行

## 8. 建议的近期执行顺序

建议按照以下顺序推进：

1. 建立测试基线
2. 引入 E2E 并打通最小回归链路
3. 收敛认证状态和数据请求模式
4. 拆分超长 repository 和重复 API 逻辑
5. 补模块文档
6. 评估数据层工具升级

## 9. 第一批落地任务清单

建议立即执行的具体任务如下：

1. 新增博客 API 测试
2. 新增消息 API 测试
3. 新增 Agent API 测试
4. 选型并接入 Playwright
5. 设计统一前端错误提示层
6. 梳理认证改造方案
7. 为 `post-repository` 和 `video-task-repository` 制定拆分方案

## 10. 结论

WitWeb 当前的主要问题不是“功能不够”，而是“质量体系和维护体系还没有跟上功能规模”。

项目已经有较好的产品雏形和基础分层，但如果继续在当前结构上直接叠加功能，后续将越来越依赖人工理解和手工回归。下一阶段最重要的工作应当是：

- 补测试
- 收敛认证和数据获取
- 统一错误处理
- 整理 API 与 repository 结构

只有先把这四件事做稳，后续扩展 Studio 和社区能力才不会持续放大维护成本。
